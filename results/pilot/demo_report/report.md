Отчет По Пилотному Эксперименту

Ниже полный разбор пилотного эксперимента на первых 50 изображениях из COCO val2017 с использованием YOLOv8n на Apple M1/MPS.

1. Цель эксперимента

Цель пилота была такой:
проверить работоспособность всего пайплайна;
оценить, как JPEG recompression и bit-depth reduction (BPC) влияют на качество детекции;
понять, есть ли смысл масштабировать эксперимент на 500 и 5000 изображений.

Эксперимент сравнивал:
baseline: исходные изображения COCO val2017;
ветка JPEG: повторное JPEG-сжатие исходных COCO JPEG с уровнями 94, 88, 75, 50, 25, 13, 6;
ветка BPC: уменьшение битовой глубины до 7, 6, 5, 4, 3, 2, 1 бит на канал с сохранением в PNG.
Важно: JPEG в этом пилоте является second-order recompression, так как исходный COCO val2017 уже распространяется в JPEG.

2. Конфигурация эксперимента

Устройство: MacBook Air M1 (2020), 8 GB RAM
Backend: PyTorch MPS
Модель: YOLOv8n
Датасет: первые 50 изображений из COCO val2017
Размер изображения: imgsz=640
Параметры валидации: batch=1, workers=0
Оцениваемые метрики:

Precision
Recall
mAP@50
mAP@50:95
Дополнительно анализировался размер папок с изображениями.

Выводы по BPC
1–2 bpc резко разрушают качество детекции.
3 bpc уже даёт заметно лучший результат, но падение качества всё ещё велико.
4 bpc выглядит как пороговая точка, где mAP50 почти совпадает с baseline.
5–7 bpc сохраняют детекцию почти на уровне оригинала, но теряют смысл с точки зрения размера файла, потому что PNG оказывается больше исходного JPEG.
Главный вывод по ветке BPC:

умеренное уменьшение битности (4–7 bpc) почти не разрушает полезную для YOLO информацию;
однако storage-efficiency у этой ветки слабая, если результат сохраняется в PNG.

5. Сравнение JPEG И BPC

Самое важное в пилоте — не отдельные ветки, а их сравнение.

Сравнение около baseline
original: 7.5 MB, mAP50 = 0.599
4 bpc: 7.9 MB, mAP50 = 0.606
q94: 5.3 MB, mAP50 = 0.596
q88: 3.8 MB, mAP50 = 0.599

Вывод:
4 bpc удерживает качество, но не даёт выигрыша по размеру;
q88 и q94 сохраняют почти тот же mAP, но занимают заметно меньше места.

Сравнение при умеренном уменьшении размера
3 bpc: 5.8 MB, mAP50 = 0.508
q94: 5.3 MB, mAP50 = 0.596
q88: 3.8 MB, mAP50 = 0.599

Вывод:
JPEG в этом диапазоне явно лучше и по размеру, и по mAP50.

Сравнение в агрессивном режиме
2 bpc: 3.1 MB, mAP50 = 0.304
q75: 2.4 MB, mAP50 = 0.587

Вывод:
q75 почти вдвое лучше по mAP50 и при этом ещё меньше по размеру.

Сравнение в экстремальном режиме
1 bpc: 1.6 MB, mAP50 = 0.187
q25: 953 KB, mAP50 = 0.496
q13: 583 KB, mAP50 = 0.285
q6: 315 KB, mAP50 = 0.101

Вывод:
q25 особенно интересен: он меньше 1 bpc, но даёт намного более высокую детекцию.
Только в области очень экстремального JPEG (q6) его качество падает ниже большинства умеренных BPC-режимов.

6. Главные Наблюдения

Наблюдение 1
Умеренное снижение bit depth действительно может сохранять полезную для YOLO информацию.

Подтверждение:

4–7 bpc дают mAP50 очень близкий к baseline.
Наблюдение 2
Однако в текущей реализации BPC + PNG не даёт хорошего trade-off по размеру.

Подтверждение:

5–7 bpc существенно больше исходного датасета;
4 bpc почти равен baseline по размеру;
только 3 bpc и ниже реально уменьшают объём, но уже заметно теряют в mAP.
Наблюдение 3
JPEG recompression оказался намного эффективнее по компромиссу size vs mAP.

Подтверждение:

q88 и q75 сохраняют почти baseline-level детекцию при радикально меньшем размере;
в сопоставимых по размеру режимах JPEG стабильно сильнее BPC + PNG.
Наблюдение 4
Метрика Precision сама по себе может быть обманчива.

Например:

q13 имеет Precision = 0.742, что выглядит высоким значением;
но Recall = 0.236 и mAP50 = 0.285, то есть модель просто делает мало предсказаний, но часть из них точная.
Вывод:

для интерпретации нужно смотреть в первую очередь на mAP50, mAP50:95 и Recall, а не только на Precision.
7. Основной Научный Вывод Пилота

Если сформулировать честно и строго, то пилот показывает следующее:

На первых 50 изображениях COCO val2017 повторное JPEG-сжатие оказалось более эффективным способом уменьшения объёма данных при сохранении качества детекции YOLOv8n, чем простое снижение битовой глубины с сохранением в PNG.

Или ещё точнее:

Умеренное снижение битовой глубины почти не разрушает task-relevant information, однако в текущей схеме хранения (PNG) оно уступает JPEG recompression по отношению между размером файла и качеством детекции.

8. Что Уже Можно Писать В Статье

На основе пилота уже можно писать раздел предварительных результатов примерно так:

A pilot experiment on the first 50 images of COCO val2017 showed that moderate bit-depth reduction preserves object detection performance reasonably well, with 4 bits per channel yielding mAP50 values close to the baseline. However, when stored in PNG format, such images do not provide a substantial storage advantage. In contrast, second-order JPEG recompression at quality levels 88 and 75 preserved detection accuracy almost at the baseline level while reducing storage size considerably. These observations suggest that, within the tested setup, JPEG recompression provides a more favorable trade-off between storage efficiency and detection performance than standalone bit-depth reduction.

9. Ограничения Пилота

Нужно честно зафиксировать ограничения:

Выборка всего 50 изображений.
Этого достаточно для пилота, но недостаточно для окончательных выводов.

Использована только одна модель:

YOLOv8n
Использован только один сценарий BPC-хранения:
bit-depth reduction + PNG
JPEG-ветка является second-order recompression, а не сжатием из lossless-источника.

Не посчитаны PSNR и SSIM, поэтому пока это анализ только по детекции и размеру.

11. Итог

Пилотный эксперимент можно считать успешным.

Он уже показал:

пайплайн работает;
метрики чувствительны к деградации;
исследовательский вопрос осмысленный;
между JPEG и BPC действительно есть содержательная разница.
Но самое важное: пилот дал не абстрактную идею, а уже конкретный промежуточный результат:

JPEG q75–q88 на этой выборке сохраняет детекцию почти на уровне baseline при гораздо меньшем размере;
BPC 4–7 сохраняет детекцию, но не даёт сопоставимой экономии по объёму;
BPC 1–2 и JPEG q13–q6 слишком разрушительны для YOLOv8n.
